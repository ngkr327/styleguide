はじめに

このスタイルガイドはチーム開発における様々な Git の使い方についてルールを明確にします。まだ手探りなところもあるので、参考情報としてご参照ください。

Git サーバーへのアクセス

git プロトコルを利用

Git サーバーからリポジトリをクローンするには次のコマンドを実行します。 

          git clone git@github.com:ngkr327/(リポジトリ名).git

    ブランチモデル

  • main

    • 開発の主軸となるブランチ。開発者は、feature ブランチで各機能の開発やバグフィックスを行い、main ブランチへマージします。

  • feature

    • 主要な機能を実装するためのブランチ。機能の実装やバグフィックスなど、タスクごとに feature ブランチを作成し開発を行います。

    ブランチ名は feature/{issue番号}_*** とします。

  • release

    • プロダクトリリースの準備のためのブランチ。リリース予定の機能やバグフィックスをほぼ反映した状態で main から分岐します。
    pre-production と同義になります。

    ブランチ名は release/vX.Y.Z とします。

  • production

    • リリースしたソースコードを管理するためのブランチ。リリース作業を行うと、production ブランチへマージされて、リリースタグが打たれます。開発者はこのブランチへのコミットは行いません。

    タグ名は vX.Y.Z とします。

    運用手順

    1. 開発準備を行う
    リポジトリ作成直後、main から production を作成します。

    2. 機能を開発する
    main から feature を作成します。feature は開発する機能毎に作成し開発していきます。

    3. 開発中の最新状態を feature に取り込む
    マージ時のコンフリクトを避けるため、任意のタイミングで main を feature へマージします。

    4. 開発済みの機能を main へ取り込む
    feature を main へマージします。Pull Request で問題ないことも確認してもらいます。

    5. リリース作業の準備を行う
    リリース準備のため、main から release を作成します。ここで、バージョンの変更や、小さなバグフィックスを行います。

    6. リリース完了
    release を production へマージし、タグを作成します。

    7. 後片付け
    issue のステータスを更新します。

    コミット

    コミットメッセージは以下とします。 

              １行目：変更内容の要約（タイトル、概要）
          ２行目：空行
          ３行目：変更した理由（内容、詳細）
  • １行目

    • コミット種別と要約を記述します。Issues, Redmine 等のチケットと連携している場合はここで紐付けを行います。
    add: 新規追加（ファイル / 機能）
    update: 機能修正
    fix: バグ修正
    remove: 削除（ファイル / 機能）

  • ３行目

    • （オプション）変更した理由をできる限り具体的に書きます。
